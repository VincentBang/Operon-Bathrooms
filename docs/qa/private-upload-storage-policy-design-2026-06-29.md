# Operon Bathrooms Private Upload Storage Policy Design - 2026-06-29

Status: POLICY_DESIGN_READY_IMPLEMENTATION_LOCKED

This document converts the existing private upload storage decision brief into an approval-ready policy design. It does
not create a Supabase Storage bucket, apply migrations, change Netlify settings, deploy, enable uploads or expose file
paths publicly.

## Current State

Current upload handling remains placeholder-safe:

- Public quote-review upload UI validates file type and size locally.
- Public API responses do not return `publicUrl`, `signedUrl`, `storagePath`, `bucket` or raw storage references.
- Uploaded file bytes are not stored by the app.
- Quote file storage is not enabled.

Existing source document:

- `docs/qa/private-upload-storage-decision.md`

## Approval Objective

Define the exact private-storage boundary that must be approved before any implementation branch can begin.

The policy is intended for quote review, request review and site measure evidence files only. It is not a customer
document vault, Quote OS file store, procurement workspace, payment workflow, proposal generator or legal/compliance
evidence system.

## Proposed Bucket Policy

Bucket name:

```text
bathroom-lead-evidence-files
```

Bucket rules:

- Private bucket only.
- No public bucket.
- No anonymous `SELECT` on `storage.objects`.
- No customer-facing object listing.
- No raw bucket/object path in public API responses.
- Short-lived signed upload/download URLs may be generated server-side only after approval.
- Service-role key must remain server-side only.

## Lead Scope

Allowed lead evidence contexts:

- `quote_review`
- `request_review`
- `site_measure`

Not allowed:

- Design Studio media storage.
- Supplier or SKU documents.
- Quote OS contract documents.
- Customer-facing document library.
- Public gallery or before/after media.
- Legal/compliance certification archive.

## Allowed File Types

Allowed MIME types:

- `application/pdf`
- `image/jpeg`
- `image/png`
- `image/webp`

Maximum file size:

- `10MB` per file for the first implementation.

Recommended file count:

- Up to `5` files per lead in the first implementation.

## Object Path Contract

Object paths must be generated server-side and must not trust customer filenames for access control.

Recommended path format:

```text
{lead_type}/{lead_id}/{upload_id}/{sanitized_filename}
```

Examples:

```text
quote_review/lead_123/upload_abc/builder-quote.pdf
request_review/lead_456/upload_def/bathroom-photo.jpg
site_measure/lead_789/upload_ghi/access-notes.png
```

Path rules:

- `lead_type` must be one of the approved lead contexts.
- `lead_id` must reference an existing lead record.
- `upload_id` must be generated by the server.
- `sanitized_filename` is display metadata only, not an authorization boundary.
- Original filename should be stored as metadata, not used as the object key without sanitization.

## Proposed Metadata Table

Future implementation may add a local migration for:

```text
bathroom_lead_evidence_files
```

Fields:

- `id uuid primary key`
- `created_at timestamptz not null default now()`
- `lead_type text not null`
- `lead_id uuid not null`
- `bucket text not null`
- `object_path text not null`
- `original_filename text not null`
- `sanitized_filename text not null`
- `mime_type text not null`
- `file_size integer not null`
- `status text not null default 'pending_upload'`
- `uploaded_by text not null default 'customer'`
- `source_route text nullable`
- `upload_context jsonb not null default '{}'::jsonb`
- `virus_scan_status text nullable`
- `internal_notes text nullable`

RLS requirements:

- RLS enabled.
- No anon `SELECT`.
- No anon `UPDATE`.
- No anon `DELETE`.
- No public policy that exposes file metadata rows.
- Service-role access only for admin reads and maintenance.
- Any anon insert or upload initiation path must be through a server-side function, not direct table access.

## Proposed Function Boundary

Future implementation should prefer server-side functions or route handlers:

- `prepare-lead-evidence-upload`
- `complete-lead-evidence-upload`
- `admin-lead-evidence-download`

Public upload preparation may return:

- Safe upload token or short-lived signed upload URL only if approved.
- Upload ID.
- Allowed MIME types.
- Maximum file size.
- Expiry timestamp.

Public upload preparation must not return:

- Supabase service-role keys.
- Bucket listing capability.
- Permanent public URLs.
- Long-lived signed URLs.
- Raw internal notes.
- Manual review report data.
- Private lead qualification fields.

Admin download may return:

- Short-lived signed download URL generated server-side.
- Safe filename/type/size metadata.

Admin download must require:

- Admin token or approved admin auth.
- Server-side service-role access only.
- Audit logging if implementation is approved.

## Public Response Boundary

Public responses may acknowledge:

- File name.
- File type.
- File size.
- Upload received or upload pending.

Public responses must never include:

- `bucket`
- `object_path`
- `storage_path`
- `publicUrl`
- `signedUrl`
- Supabase storage host URLs.
- Admin notes.
- Manual review reports.
- Lead scoring or qualification internals.

## QA Gates Before Implementation

Before implementation begins:

- Approval must confirm this policy design.
- Approved local or staging Supabase target must be named.
- Production Supabase must remain untouched.
- Production Netlify must remain untouched.

Before merge of any implementation:

- `npm run qa:local`
- `npm run qa:public-safety -- http://127.0.0.1:3000`
- `npm run qa:bundle-safety`
- `npm run verify:supabase:migrations`
- `npm run qa:supabase:staging` against approved non-production target only.
- Public API safety tests proving fake/public/signed/storage URL inputs are not echoed.
- Admin-only retrieval tests.
- Migration verifier catches anon `storage.objects` policies.

Manual QA:

- Public form upload UI remains usable on mobile.
- Failed upload states do not expose storage internals.
- Admin file retrieval requires admin access.
- No public page, sitemap, robots or chatbot response exposes storage paths.
